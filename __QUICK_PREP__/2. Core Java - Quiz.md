## OOPS

-	** What is the difference between Functional Programming, OOPS, and AOP ? **
	-	OOPS (Object-Oriented Programming): Focuses on Objects and Classes. It uses encapsulation, inheritance, and polymorphism to organize code around data. It is stateful and relies on modifying object properties.
	-	Functional Programming (FP): Focuses on Functions and Immutability. It treats computation as the evaluation of mathematical functions and avoids changing-state and mutable data. It uses "Pure Functions" where the output depends only on the input.
	-	AOP (Aspect-Oriented Programming): Focuses on Cross-cutting Concerns. It complements OOPS by allowing you to separate secondary tasks (like logging, security, or transaction management) from the main business logic using "Aspects."
	-	Key Distinction: OOPS is about nouns (Objects), FP is about verbs (Functions), and AOP is about concerns that affect multiple parts of the application.

## Java 8 Features

-	** What are Functional Interfaces in Java 8 ? **
	-	Definition: An interface that contains exactly one abstract method is known as a functional interface. It can have any number of default or static methods.
	-	@FunctionalInterface: An optional annotation used to ensure the compiler enforces the single abstract method rule, preventing accidental additions of other abstract methods. it will throw an error if a second abstract method is added.
	-	Purpose: It allows you to treat functionality as an argument, making code more concise and readable by eliminating the need for bulky anonymous inner classes enabling functional programming in Java.
	-	Core Built-in Interfaces: Java 8 introduced several standard ones in the java.util.function package, such as Predicate (returns boolean), Function (transforms input), Consumer (accepts input, returns nothing), and Supplier (provides a value).
	-	Example: Runnable, callable

-	** What is the difference between a Functional Interface and a standard @Annotation , and what changed after Java 8 ? **
	-	In short, Functional Interface is the structure, and @FunctionalInterface annotation is the compiler-enforced annotation certifying that structure
	-	Functional Interface: A Java interface with exactly one abstract method used as a target for Lambda Expressions. It defines behavior that can be passed around as code.
	-	@Annotation: A form of metadata used to provide data about a program that is not part of the program itself. It defines attributes used by the compiler or runtime tools for processing (e.g., @Override, @Entity).
	-	Changes after Java 8:
		-	Lambda Support: Functional interfaces became the backbone for the Stream API, allowing developers to write compact, functional-style code instead of verbose anonymous classes.
		-	Method References: Java 8 allowed using the :: operator to point to existing methods as implementations of functional interfaces.
		-	Default/Static Methods: Interfaces can now contain concrete methods, allowing functional interfaces to provide helper logic without breaking the "single abstract method" rule.
		-	Private Methods (Java 9): Functional interfaces can now contain private methods to encapsulate logic shared between default methods.

-	** What is the difference between Consumer, Predicate, Supplier, and Function in Java 8 ? **

	-	Bi-Variants: Specialized versions like BiPredicate<T, U>, BiFunction<T, U, R>, and BiConsumer<T, U> that accept two input arguments instead of one

-	** What is the difference between Runnable and Comparator as Functional Interfaces ? **
	-	Runnable: Contains the run() method; it takes no arguments and returns void. It is primarily used to define a task that can be executed by a thread or an ExecutorService.
	-	Comparator<T>: Contains the compare(T o1, T o2) method; it takes two arguments of the same type and returns an int (negative, zero, or positive). It is used to define custom sorting order for collections or arrays.

		// Using Lambda to implement Runnable
		// Used to execute a block of code, often in a separate thread. It takes no parameters and returns nothing.

		Thread thread = new Thread(task);
		thread.start();
		```

		// Used to define the sorting order. It takes two parameters and returns an integer result
		List<String> names = Arrays.asList("Java", "C", "Python", "Go");

		Comparator<String> lengthComparator = (s1, s2) -> Integer.compare(s1.length(), s2.length());

-	** What is the difference between implementing an Interface in a Class versus using a Functional Interface with a Lambda ? **
	-	Class Implementation: Requires a separate class or an anonymous inner class to provide the method logic. It creates a physical .class file and can maintain internal state (instance variables).
	-	Lambda Implementation: Provides an inline implementation of the single abstract method without the need for a separate class. It is more concise and focuses solely on the behavior (logic) rather than the structure.
	-	Memory: Lambda expressions are generally more memory-efficient as they do not always create a new object instance (using invokedynamic) compared to the overhead of creating a full class instance.
	-	Flexibility: Class implementation is better for complex logic requiring multiple methods or fields, while Lambdas are ideal for short, one-time tasks like filtering or mapping in a stream.

-	** What is the Stream API in Java 8 ? **
	-	Definition: A powerful framework in the java.util.stream package used to process sequences of elements (collections, arrays) in a functional and declarative style.
	-	Key Characteristics: It does not store data (it's a view, not a collection), does not modify the source, and supports lazy evaluation (operations aren't executed until a terminal result is needed).
	-	Core Workflow: Consists of a Source (e.g., list.stream()), followed by zero or more Intermediate Operations (e.g., .filter(), .map(), .distinct()), and exactly one Terminal Operation (e.g., .collect(), .forEach(), .count()).
	-	Parallelism: Provides an effortless way to achieve multi-core processing by simply calling .parallelStream() instead of .stream()

-	** What is the difference between Parallel and Sequential Streams in Java ? **
	-	Sequential Stream: The default stream() that processes elements in a single thread, ensuring a predictable, linear execution order.
	-	Parallel Stream: Created via parallelStream(), it splits the source data into multiple chunks using the Fork/Join framework and processes them simultaneously across multiple CPU cores.
	-	Performance: Parallel streams are generally faster for very large datasets and computationally heavy tasks, but can be slower for small tasks due to the overhead of thread management and merging results.
	-	Thread Safety: Parallel streams must be used with caution; the operations must be stateless and non-interfering to avoid race conditions when accessing shared resources.

-	** What are the common methods used in the Java 8 Stream API ? **
	-	map(): An intermediate operation that uses a Function to transform each element into a new value (e.g., converting a list of strings to their lengths).
	-	filter(): An intermediate operation that uses a Predicate to include only elements that match a specific condition (e.g., keeping only numbers greater than 10).
	-	toArray(): A terminal operation that converts the resulting stream elements into a physical array (e.g., String[] arr = stream.toArray(String[]::new)).
	-	collect(): A terminal operation that performs a mutable reduction, allowing you to transform the stream back into a collection like a List, Set, or Map.
	-	Collectors: A utility class providing pre-built implementation for collect(), such as toList(), toSet(), and joining().

-	** What is the difference between map and filter in the Stream API ? **
	-	filter(): An intermediate operation that takes a Predicate (a functional interface with a boolean test(T t) method) to include or exclude elements based on a condition.
	-	map(): An intermediate operation that takes a Function (a functional interface with an R apply(T t) method) to transform each element into another object or value.

-	** What is the difference between map and filter in terms of Stream operations and efficiency ? **
	-	filter(): An intermediate operation that reduces the number of elements in a stream by keeping only those that satisfy a given boolean condition (Predicate). It reduces processing load for subsequent steps.
	-	map(): An intermediate operation that transforms each element into a new form (Function) while maintaining the original count of elements in the stream.
	-	Memory Efficiency: Both methods are Lazy; they do not create new intermediate collections or copies of the array in memory. They process elements one by one only when a terminal operation is called.
	-	Processing Performance: filter() is often used before map() to minimise memory and CPU usage, ensuring that heavy transformations are only performed on the necessary subset of data.

-	** What is the difference between map and flatMap in the Stream API ? **
	-	map(): An intermediate operation that transforms each element into exactly one other element. Use it for 1-to-1 transformations (e.g., converting [1, 2, 3] to ["1", "2", "3"]).
	-	flatMap(): An intermediate operation that transforms each element into a stream of elements and then "flattens" them into a single resulting stream.
	-	Flattening Mechanism: It takes a nested structure, such as [[1,2,3], [4,5,6], [7]], and collapses it into a single-level sequence: [1, 2, 3, 4, 5, 6, 7].
	-	Use Case: Use flatMap() when you have an array inside another array (or a list of lists) and you want to process all internal elements as a single, continuous stream.

-	** What is Parallel Array Sorting ? **
	-	Definition: A feature introduced in Java 8 via the Arrays.parallelSort() method that uses the Fork/Join framework to sort an array in parallel across multiple CPU cores.
	-	Mechanism: The array is recursively divided into sub-arrays (the "Fork" phase) until they reach a threshold size, at which point they are sorted sequentially using Dual-Pivot Quicksort or TimSort; finally, the sorted sub-arrays are merged back together (the "Join" phase).
	-	Performance: It is significantly faster than the standard Arrays.sort() for large arrays (typically elements) on multi-core systems, but it may have higher overhead for very small arrays due to thread management.
	-	Threshold: The JVM automatically decides the threshold for parallelism based on the available processors and the array size to ensure optimal resource usage.

-	** What are Type Annotations and Repeating Annotations in Java 8 ? **
	-	Type Annotations: An expansion of the annotation system that allows annotations to be applied anywhere a type is used, such as in generic type arguments (List<@NonNull String>), object allocations (new @Interned MyObject()), or type casts. This enables stronger static analysis and bug detection by external frameworks like the Checker Framework.
	-	Repeating Annotations: A feature that allows the same annotation to be applied multiple times to the same declaration (e.g., @Schedule(day="Mon") @Schedule(day="Fri")). It requires a "Container Annotation" to hold the repeated values and the use of the @Repeatable meta-annotation on the original annotation.
	-	Improved Metadata: These features provide developers with more expressive power to describe code constraints and configurations without needing complex XML or separate metadata files. **

-	** What is the improvement in HashMap before and after Java 8 ? **
	-	Before Java 8: Used Singly Linked Lists to handle collisions. In the worst case (many collisions in one bucket), the time complexity degraded from O(1) to O(n)
	-	After Java 8: Introduced Treeification. When a bucket exceeds a threshold of 8 elements, the linked list is replaced by a Red-Black Tree (Balanced Tree)
	-	Benefit: This optimization ensures that even with many collisions, the worst-case lookup complexity is improved from O(n) to O(log n)

-	** How do you remove duplicates from a List in Java 8 ? **
	-	Implementation: Use the Stream API by calling .stream(), followed by the .distinct() intermediate operation, and finally collect the results back into a new list using .collect(Collectors.toList()).
	-	```java 
		List<String> uniqueList = list.stream().distinct().collect(Collectors.toList());
		```
	-	Efficiency: It uses the hashCode() and equals() methods of the objects to identify and remove duplicates while maintaining the original order of elements.

-	** What is the difference between the collect() method and the Collectors class in the Stream API ? **
	-	collect(): A terminal operation of the Stream interface that performs a mutable reduction on the elements. It acts as the "container" or the action that triggers the processing to produce a final result.
	-	Collectors: A utility class that provides many static factory methods (like toList(), toSet(), joining(), or groupingBy()) which implement common reduction strategies.
	-	Relationship: You pass a implementation from the Collectors class as an argument into the collect() method to define how the stream should be gathered into a collection or value.
	-	Example: In list.stream().collect(Collectors.toList()), collect is the action being performed, and Collectors.toList() is the specific instruction for that action.

-	** What is the difference between Program Arguments and VM Arguments ? ** 
	-	Program Arguments: These are passed to the main(String[] args) method of your application and are used to control application-specific logic (e.g., file paths, usernames, or processing flags).
		-	How to provide: Added after the class name in the command line (e.g., java MyClass arg1 arg2).
	-	VM Arguments (System Properties): These are used to configure the Java Virtual Machine (JVM) environment and behavior, such as memory limits, garbage collection, or system properties.
		-	How to provide: Prefixed with -D or -X and added before the class name (e.g., java -Xmx512m -Denv=prod MyClass)
	
-	** Why main method in java is public static void ? **
	-	public: The JVM (which is an external program) needs to access this method to start the application. If it were private or protected, the JVM wouldn't have the permission to "see" or execute it.
	-	static: The JVM needs to call the method without creating an instance (object) of the class first. Since the main method is the starting point, no objects exist yet when the program begins.
	-	void: Once the main method finishes, the program terminates. There is no other Java method "waiting" for a response, so it doesn't need to return any data.
	
-	** What is the use of String args[] in main method ? **
	-	Command-Line Inputs: It allows the user to pass data into the program at the moment it is launched from the terminal or command prompt.
	-	Array Storage: The JVM collects all arguments provided after the class name and stores them as an array of Strings, where args[0] is the first argument.
	-	Configuration: It is commonly used to pass file paths, environment names (like dev or prod), or configuration flags without hardcoding them into the source code.

-	** What is the difference between the hashCode() and equals() methods ? **
	-	Purpose: The equals() method determines if two objects are "meaningfully equivalent" based on their content, while hashCode() returns an integer value representing the object's memory address or content for use in hashing-based collections like HashMap.
	-	Contract: If two objects are equal according to equals(), they must have the same hashCode(). However, two objects with the same hashCode() are not necessarily equal (this is known as a collision).
	-	Usage in Collections: When adding an object to a HashSet or HashMap, Java first checks the hashCode() to find the correct bucket; it only calls equals() if there is a hash collision in that bucket.
	-	Overriding: If you override one, you must override the other to maintain the contract, otherwise, hashing-based collections will fail to retrieve or store your objects correctly.
	
-	** How does HashMap internally work ? **
	-	Core Structure: It uses an internal array of buckets (initially 16 by default). Each bucket stores a LinkedList or a Red-Black Tree of nodes containing the key, value, and hash.
	-	Hashing and Indexing: When put(key, value) is called, the key’s hashCode() is used to calculate a hash value, which is then mapped to a specific bucket index using the formula: index = hash and (n -	1).
	-	Collision Handling: If two different keys land in the same bucket (collision), they are stored in a LinkedList.
	-	Treeification (Java 8+): If a bucket's list exceeds 8 elements (and the total capacity is at least 64), it converts the list into a Red-Black Tree to improve search time from O(n) to O(log n).
	-	Retrieval: To get(key), the index is recalculated; Java then traverses the bucket and uses equals() to find the exact key-value match.
	-	Resizing: When the number of entries exceeds the load factor threshold (default 0.75), the array capacity doubles, and all entries are rehashed into the new array.

-	** How do you create and handle a Java Custom Exception ? **
	-	Creation: Create a class that extends Exception (for a checked exception) or RuntimeException (for an unchecked exception). It is best practice to provide a constructor that accepts a String message.

-	** What are the rules for Exception Handling in Method Overriding ? **
	-	If Superclass method does not declare an exception: The overriding subclass method cannot declare any new or broader checked exceptions. However, it is free to declare any unchecked exceptions (RuntimeExceptions).
	-	If Superclass method declares an exception: The overriding subclass method can:
		-	Declare the same exception as the superclass.
		-	Declare a subclass (narrower) of that exception (e.g., throwing FileNotFoundException if the parent throws IOException).
		-	Declare no exception at all.
		-	Cannot declare a parent (broader) exception (e.g., throwing Exception if the parent throws IOException).
	-	Unchecked Exceptions: There are no restrictions for unchecked exceptions; a subclass can always throw them regardless of what the superclass method declares.
	-	Reasoning: These rules ensure the Liskov Substitution Principle, preventing child classes from breaking the "contract" established by the parent class which callers rely on

-	** What is the volatile keyword ? **
	-	Visibility: It ensures that changes made to a variable by one thread are immediately visible to all other threads by reading from and writing directly to main memory instead of the CPU cache.
	-	Atomicity: It does not provide atomicity; for compound operations like count++ (which is read-modify-write), volatile is not enough and you should use synchronized or AtomicInteger.
	-	Instruction Reordering: It prevents the JVM from reordering instructions around the volatile variable, ensuring that any code written before the volatile write stays before it.
	-	Best Use Case: It is ideal for boolean flags or status indicators (e.g., a stop flag) where one thread modifies the value and others only need to see the most recent update.

-	** What is the difference between a process and a thread ? **
	-	Process: An independent unit of execution with its own separate memory space (heap and stack) and system resources. A program becomes a process when it is executed (e.g., launching the JVM).
	-	Thread: A lightweight, executable path within a process. Multiple threads inside one process share the same heap memory and data segments but maintain their own private stack and program counter.
	-	Communication: Processes require complex Inter-Process Communication (IPC) to exchange data. Threads communicate directly through shared memory, which is faster but requires synchronization to prevent race conditions.
	-	Overhead: Creating and switching between processes is resource-intensive ("heavyweight") because it involves allocating new memory blocks. Threads are "lightweight" and faster to create or switch because they reuse existing process resources.
	-	Failure Impact: If a process crashes, it generally does not affect other processes. If a thread crashes or misbehaves, it can potentially bring down the entire process and all other sibling threads.
	
-	** What are the basics of threads in Java ? **
	-	Thread Creation: You can create a thread by either extending the Thread class and overriding run() or by implementing the Runnable interface and passing it to a Thread constructor (preferred for flexibility).
	-	Execution: To start a thread, you must call the start() method, which allocates system resources and triggers the JVM to call the run() method; calling run() directly executes it in the current thread like a normal method.
	-	Thread States: A thread moves through various lifecycle stages: New (created), Runnable (ready to run), Running (executing), Blocked/Waiting (suspended for resources), and Terminated (finished).
	-	Main Thread: Every Java program has at least one thread, the main thread, which is automatically created by the JVM to execute the public static void main method.
	-	Properties: Each thread has a Priority (1 to 10), a Name for debugging, and can be marked as a Daemon thread (runs in the background and doesn't prevent the JVM from exiting).

-	** What are the different ways to create and use threads in Java ? **
	-	Extending the Thread class: You create a new class that extends Thread and overrides the run() method; you then instantiate that class and call .start().
		-	Drawback: Since Java doesn't support multiple inheritance, your class cannot extend any other class if it already extends Thread.
	-	Implementing the Runnable interface: You define a class that implements Runnable, provide the run() logic, and pass an instance of it to a Thread object.
		-	Advantage: This is the preferred approach because it allows your class to extend another class and separates the task logic from the thread control.
	-	Using Lambda Expressions (Java 8+): Since Runnable is a functional interface, you can start a thread concisely without creating a separate class using new Thread(() -> { /* logic */ }).start();.
	-	Using ExecutorService: Instead of manually managing Thread objects, you use a Thread Pool (e.g., Executors.newFixedThreadPool()) to submit Runnable or Callable tasks for more efficient resource management.

-	** What is Garbage Collection in Java ? **
	-	Definition: Garbage Collection (GC) is an automatic memory management process in the Java Virtual Machine (JVM) that identifies and deletes objects no longer in use to reclaim heap space.
	-	Generational Approach: Most objects are short-lived, so the heap is divided into segments:
		-	Young Generation: Where new objects are created; it is subdivided into Eden and two Survivor spaces.
		-	Old (Tenured) Generation: Stores long-lived objects that survived multiple minor GC cycles.
	-	Core Phases:
		-	Mark: The GC identifies all live objects by traversing the object graph starting from "GC Roots" (like static variables and active threads).
		-	Sweep: Reclaims memory from unmarked (dead) objects.
		-	Compact: Rearranges live objects to eliminate fragmentation and create contiguous free space.
	-	Modern Collectors:
		-	G1 (Garbage-First): The default since Java 9, it divides the heap into regions and prioritizes those with the most garbage for predictable pauses.
		-	ZGC / Shenandoah: Modern, ultra-low-latency collectors that perform most work concurrently with application threads, keeping pause times under 10ms even on massive heaps
	-	Practice articulating: "In production, I tuned G1 to reduce p99 latency from 500ms to 50ms by region sizing and humongous allocations."
	
-	** What is the difference between p* (Percentile) latencies ? **
	-	p50 (Median): The response time for which 50% of requests are faster and 50% are slower. It represents the "typical" user experience but can hide extreme slowness.
	-	p90: The threshold where 90% of requests are faster, and the slowest 10% experience performance worse than this value.
	-	p95: The value where 95% of requests are completed within this time; it is a common benchmark for standard production systems to catch most outliers.
	-	p99 (Tail Latency): The threshold where 99% of requests are faster, and only the slowest 1% exceed it. This is a critical metric for large-scale systems because, at millions of requests, even 1% affects thousands of users.
	-	p99.9: Used in ultra-sensitive or mission-critical systems (like banking or high-frequency trading) where even 0.1% of slow requests can lead to significant issues or financial loss.

-	** What is the difference between Multithreading and Concurrency ? **
	-	Multithreading: A specific form of concurrency where a single process is divided into multiple threads that execute independently while sharing the same memory and resources. It is a way to achieve multitasking within a single application.
	-	Concurrency: The ability of a system to handle multiple tasks at the same time by interleaving their execution. It is a broad concept; tasks may start, run, and complete in overlapping time periods without necessarily running at the exact same instant.
	-	Concurrency vs. Parallelism: Concurrency is about dealing with many things at once (managing structure), while Parallelism is about doing many things at once (requiring multiple CPU cores).
	-	In Java: Multithreading is the tool (using the Thread class or ExecutorService), and Concurrency is the goal (designing the application to perform multiple operations simultaneously).

-	** What is Deadlock in Java ? **
	-	Definition: Deadlock is a situation where two or more threads are blocked forever, each waiting for a resource held by the other, creating a circular dependency that stops execution.
	-	The "Deadly Embrace" Example:
		-	Thread A holds Lock 1 and waits for Lock 2.
		-	Thread B holds Lock 2 and waits for Lock 1.
		-	Neither thread can proceed, and the program hangs indefinitely.
	-	Necessary Conditions: 
		-	For a deadlock to occur, four conditions must be met: 
		-	Mutual Exclusion (locks), 
		-	Hold and Wait (holding one lock while waiting for another), 
		-	No Preemption (locks can't be forced away), 
		-	and Circular Wait.
	-	How to Avoid:
		-	Lock Ordering: Always acquire multiple locks in the same consistent order across all threads.
		-	Lock Timeout: Use tryLock() from the ReentrantLock class with a timeout instead of synchronized blocks.
		-	Minimize Locking: Keep synchronized blocks as short as possible and avoid nested locks whenever possible.

-	** What is the life cycle of a thread in Java ? **
	> https://www.geeksforgeeks.org/java/lifecycle-and-states-of-a-thread-in-java/
	-	New: A thread is in this state when an instance of the Thread class is created but the start() method has not yet been called.
	-	Runnable: After calling .start(), the thread is ready to run and is waiting for the thread scheduler to allocate CPU time. It could be currently running or waiting in the queue.
	-	Blocked: A thread enters this state when it is alive but currently waiting to acquire a monitor lock to enter a synchronized block or method.
	-	Waiting: A thread is in this state if it is waiting indefinitely for another thread to perform a specific action (e.g., via object.wait() or thread.join()).
	-	Timed Waiting: Similar to the waiting state, but the thread waits only for a specified period (e.g., Thread.sleep(1000) or object.wait(1000)).
	-	Terminated (Dead): The thread reaches this state when it has finished executing its run() method or if it was stopped due to an unhandled exception.

-	** What is the join() method in Java threads ? **
	-	Purpose: The join() method allows one thread to wait for the completion of another thread. If you call threadA.join() inside the main thread, the main thread will pause and wait until threadA finishes its execution.
	-	Synchronization: It is used to ensure a specific execution order when one thread's result is needed by another (e.g., waiting for a data-fetching thread to finish before processing the data).
	-	InterruptedException: Since join() puts the calling thread into a Waiting state, it throws a checked InterruptedException, which must be handled with a try-catch block.
	-	Timed Join: You can also use threadA.join(1000), which tells the calling thread to wait for a maximum of 1000 milliseconds. If the thread isn't finished by then, the calling thread resumes anyway.
	
	```java
	
		Thread worker = new Thread(() -> {
	    try {
	        System.out.println("Worker: Working...");
	        Thread.sleep(2000); // Simulate task
	        System.out.println("Worker: Task Complete.");
	    } catch (InterruptedException e) {
	        e.printStackTrace();
	    }
	});

	worker.start(); // Start the worker thread

	try {
	    System.out.println("Main: Waiting for worker...");
	    worker.join();  // Main thread pauses here until worker is dead
	    System.out.println("Main: Worker finished, resuming main task.");
	} catch (InterruptedException e) {
	    e.printStackTrace();
	}
	
	```

-	** What is a Daemon Thread in Java ? **
	-	Definition: A daemon thread is a low-priority thread that runs in the background to perform tasks such as garbage collection or house-keeping for the JVM.
	-	Lifecycle: The JVM does not wait for daemon threads to finish before exiting. If all user threads (non-daemon threads) complete their execution, the JVM will shut down automatically, even if daemon threads are still running.
	-	Creation: You can mark a thread as daemon by calling setDaemon(true) on a thread instance before calling the start() method.
	-	Inheritance: A thread created by a daemon thread is itself a daemon thread by default.
	-	Usage: They are ideal for "service" tasks that should not prevent the application from closing, such as a background log cleaner or a resource monitor.

-	** What is a Thread Pool in Java ? **
	-	Definition: A Thread Pool is a collection of pre-instantiated, reusable threads managed by the Executor Framework to perform tasks asynchronously.
	-	Efficiency: Instead of creating a new thread for every task (which is expensive in terms of CPU and memory), it reuses existing threads, reducing the overhead of thread creation and destruction.
	-	Resource Management: It prevents the system from crashing under heavy load by limiting the maximum number of active threads running at any given time.
	-	Components:
		-	Task Queue: Holds incoming tasks (Runnables/Callables) until a thread becomes available.
		-	Worker Threads: Threads that pull tasks from the queue and execute them.
		-	ExecutorService: The primary interface used to submit tasks and manage the pool's lifecycle.
	-	Common Types (Executors class):
		-	FixedThreadPool: A pool with a fixed number of threads.
		-	CachedThreadPool: Creates new threads as needed but reuses old ones; ideal for many short-lived tasks.
		-	ScheduledThreadPool: Used to execute tasks after a delay or periodically.
		-	SingleThreadExecutor: Uses only one worker thread to process tasks sequentially.
		
-	** Can you provide a code example for a FixedThreadPool ? **
	-	Implementation: You use the Executors utility class to create a FixedThreadPool with a specific number of threads, then submit Runnable tasks to it.
	-	Behavior: In below example, 3 tasks will run concurrently while the remaining 2 wait in a queue until a worker thread becomes free
	```java
		import java.util.concurrent.ExecutorService;
		import java.util.concurrent.Executors;

		public class FixedThreadPoolExample {
			public static void main(String[] args) {
				// Create a pool with 3 worker threads
				ExecutorService executor = Executors.newFixedThreadPool(3);

				// Submit 5 tasks to the pool
				for (int i = 1; i <= 5; i++) {
					int taskId = i;
					executor.submit(() -> {
						String threadName = Thread.currentThread().getName();
						System.out.println("Task " + taskId + " executed by " + threadName);
						try { Thread.sleep(1000); } catch (InterruptedException e) {}
					});
				}

				// Shut down the pool to stop accepting new tasks
				executor.shutdown(); 
			}
		}

	```
	
-	** How do you use Future to get results from a Thread Pool ? **
	-	Callable Interface: Unlike Runnable, the Callable<V> interface allows a task to return a result and throw checked exceptions.
	-	Future Object: When you submit a Callable to an ExecutorService, it returns a Future<V> object, which acts as a "placeholder" for the result that will be available later.
	-	Blocking Retrieval: Calling future.get() will block the current thread's execution until the background task is finished and the result is ready.
	-	Checking Status: You can use future.isDone() to check if the task is finished without blocking, or future.cancel(true) to attempt to stop the task if it's no longer needed.
	```java
		ExecutorService executor = Executors.newFixedThreadPool(2);
	
		// Submit a task that returns a String
		Future<String> future = executor.submit(() -> {
			Thread.sleep(2000); // Simulate long task
			return "Task Result Data";
		});
	
		// Do other work here...
		System.out.println("Doing other work in main thread...");
	
		try {
			// This blocks until the result is ready
			String result = future.get(); 
			System.out.println("Received: " + result);
		} catch (Exception e) { e.printStackTrace(); }
	
		executor.shutdown();
	
	```
	
-	** How does CompletableFuture improve on the Future interface ? **
	-	Non-blocking Callbacks: Unlike Future.get(), which blocks the thread, CompletableFuture allows you to attach callbacks like .thenAccept() or .thenApply() that execute automatically as soon as the result is ready.
	-	Chaining Tasks: You can easily chain multiple asynchronous steps together (e.g., fetch data → transform data → save to DB) using methods like .thenCompose().
	-	Exception Handling: It provides a functional way to handle errors using .exceptionally(), allowing you to provide a default value or recovery logic if a background task fails.
	-	Combining Futures: You can wait for multiple independent futures to complete simultaneously using CompletableFuture.allOf() or pick the first one to finish with anyOf().
	```java
		CompletableFuture.supplyAsync(() -> {
			// Step 1: Simulate heavy calculation
			try { Thread.sleep(2000); } catch (InterruptedException e) {}
			return "Raw Data";
		})
		.thenApply(data -> data + " -> Processed") // Step 2: Transform
		.thenAccept(result -> System.out.println("Final Output: " + result)) // Step 3: Use result
		.exceptionally(ex -> {
			System.out.println("Error occurred: " + ex.getMessage());
			return null;
		});

		// Main thread continues immediately without blocking!
		System.out.println("Main thread is free to do other work...");

	```
	
-	** How do you use a custom Executor with CompletableFuture ? **
	-	Thread Control: By default, CompletableFuture uses the ForkJoinPool.commonPool(), which is shared across the entire JVM. Passing a custom Executor allows you to isolate your tasks into a dedicated thread pool to prevent resource exhaustion.
	-	Overloaded Methods: Almost all CompletableFuture methods (like supplyAsync, runAsync, thenApplyAsync) have an overloaded version that accepts an Executor as the second argument.
	-	Best Practice: Use Async suffix methods (e.g., thenAcceptAsync) when the callback logic is heavy, as it ensures the next step also runs in your controlled thread pool rather than the thread that completed the previous stage.
	
	```java
		// Define a dedicated thread pool for I/O tasks
		ExecutorService customExecutor = Executors.newFixedThreadPool(5, r -> {
			Thread t = new Thread(r);
			t.setName("IO-Worker-" + t.getId());
			return t;
		});
	
		// Pass the custom executor to the async task
		CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
			System.out.println("Running in: " + Thread.currentThread().getName());
			return "Data fetched from API";
		}, customExecutor);
	
		future.thenAcceptAsync(result -> {
			System.out.println("Processing " + result + " in: " + Thread.currentThread().getName());
		}, customExecutor);
	
		customExecutor.shutdown();
	
	```

-	** What is the difference between handle() and exceptionally() in CompletableFuture ? **
	-	exceptionally(): This is used solely for error recovery. It only executes if the preceding stage throws an exception, allowing you to return a "fallback" or default value.
	-	handle(): This is a universal method that always executes, regardless of whether the previous stage succeeded or failed. it receives two arguments: the result (if successful) and the exception (if it failed).
	-	Flexibility: handle() is more powerful because it allows you to transform both the successful result and the error in one place, whereas exceptionally() only deals with the error.
	
	```java
		CompletableFuture.supplyAsync(() -> {
			if (Math.random() > 0.5) throw new RuntimeException("API Error");
			return "Success Data";
		})
		// 'handle' runs in BOTH cases
		.handle((res, ex) -> {
			if (ex != null) {
				return "Recovered from: " + ex.getMessage();
			}
			return "Processed: " + res;
		})
		.thenAccept(System.out::println);

	```

-	** What is the difference between CompletableFuture.allOf() and anyOf() ? **
	-	allOf(): Creates a new future that completes only when every provided future has finished. It returns CompletableFuture<Void>, meaning it doesn't directly return the results; you must retrieve them individually from the original futures using methods like .join() after completion.
	-	anyOf(): Returns a new future that completes as soon as any one of the provided futures finishes (whether successfully or with an exception). It returns CompletableFuture<Object>, containing the result of the fastest task.
	-	Use Case for allOf(): Best for "fan-in" scenarios where you need multiple independent results (e.g., fetching data from three different APIs) before proceeding to a final aggregation step.
	-	Use Case for anyOf(): Ideal for "fastest-wins" scenarios, such as querying multiple redundant servers for the same data and taking the first response to minimize latency** What is the difference between CompletableFuture.allOf() and anyOf() ? **

	```java

	CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> "Task 1");
	CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> "Task 2");

	// allOf example
	CompletableFuture.allOf(f1, f2).thenRun(() -> {
		System.out.println("Both done: " + f1.join() + ", " + f2.join());
	});

	// anyOf example
	CompletableFuture.anyOf(f1, f2).thenAccept(result -> {
		System.out.println("Fastest result: " + result);
	});

	```
	
-	** How do you handle partial failures with CompletableFuture.allOf() ? **
	-	 The allOf() Limitation: By design, CompletableFuture.allOf() returns Void and only completes after all provided futures have finished, regardless of whether they succeeded or failed. If one fails, the combined future will eventually throw an exception when joined, but only after waiting for every other task to finish.
	-	 Fault Isolation (Individual Handling): To handle partial failures, you must attach exceptionally() or handle() to each individual future before passing them to allOf(). This allows you to catch errors for specific tasks and return a default/null value, ensuring the other tasks can still complete and contribute to the final result.
	-	 Manual Result Aggregation: Since allOf() returns Void, you must manually collect results from the original list of futures after completion. You can use a stream to map over the futures and call .join() or .getNow(null) to extract the successful results or the fallback values you provided during individual error handling.
	-	 Fail-Fast Strategy: If you want the entire process to stop immediately when the first failure occurs (rather than waiting for all others), you must manually complete the combined future using completeExceptionally() inside an individual task's error handler.

	```java
		CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> "Data 1");
		CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> {
			throw new RuntimeException("Service 2 Down"); 
		}).exceptionally(ex -> null); // Handle failure locally
		CompletableFuture<String> f3 = CompletableFuture.supplyAsync(() -> "Data 3");

		List<CompletableFuture<String>> futures = Arrays.asList(f1, f2, f3);

		// Wait for all to complete
		CompletableFuture<Void> allDone = CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]));

		allDone.thenRun(() -> {
			List<String> results = futures.stream()
				.map(CompletableFuture::join)
				.filter(Objects::nonNull) // Ignore the "null" from f2
				.collect(Collectors.toList());

			System.out.println("Successful results: " + results); 
			// Output: [Data 1, Data 3]
		}).join();

	```

		})

-	** What is a Synchronized Block in Java ? **
	-	Purpose: It is a block of code used to lock a specific object or resource, ensuring that only one thread can execute that section of code at a time to prevent data inconsistency.
	-	Granularity: Unlike a synchronized method (which locks the entire object), a synchronized block allows you to lock only a specific portion of code, making it more efficient and reducing thread contention.
	-	Lock Object: It requires a reference to an object (the "monitor lock"). You can lock on this (the current instance), a specific object variable, or the class itself (e.g., MyClass.class) for static synchronization.
	-	Wait/Notify: Synchronized blocks provide the foundation for inter-thread communication using wait(), notify(), and notifyAll(), which must be called from within a synchronized context.
	```java
		synchronized (lockObject) {
			// Critical section: only one thread at a time
		}
	```
	
-	** What is the difference between a Synchronized block and ReentrantLock ? **
	-	Key Advantage of Synchronized: It is simpler to use and less error-prone because the JVM automatically manages lock acquisition and release, even if an exception occurs.
	-	Key Advantage of ReentrantLock: It provides advanced features like tryLock(), which allows a thread to attempt to acquire a lock without waiting indefinitely, helping to avoid deadlocks.
	-	Virtual Thread Note: In Java versions 21 to 23, synchronized can cause thread pinning, which limits scalability; replacing it with ReentrantLock is a common fix for virtual threads in those versions.
	
	| Feature               | synchronized Block                               | ReentrantLock (java.util.concurrent.locks)                |
	|------------------------|--------------------------------------------------|-----------------------------------------------------------|
	| **Type**               | Implicit (built-in keyword)                      | Explicit (library class in `java.util.concurrent.locks`)  |
	| **Lock Release**       | Automatic (handled by JVM)                       | Manual (requires `unlock()` in a `finally` block)         |
	| **Flexibility**        | Rigid; one thread either gets the lock or blocks | High; supports `tryLock()` with timeouts and interruption |
	| **Fairness**           | No; waiting order is not guaranteed              | Optional; fairness policy can prioritize longest-waiting thread |
	| **Condition Variables**| Single wait-set (`wait()`, `notify()`, `notifyAll()`) | Multiple condition variables per lock (`newCondition()`)  |
	| **Performance**        | Optimized by JVM for low contention              | Better under high contention; more control for tuning     |
	| **Interruptible Locking** | No (thread cannot be interrupted while waiting) | Yes (`lockInterruptibly()` allows interruption)           |
	| **Reentrancy**         | Yes (thread can reacquire the same lock)         | Yes (supports reentrancy explicitly)                      |

	```java
		public void synchronizedUpdate() {
			synchronized (this) {
				// Critical section
				System.out.println("Locked with synchronized");
			}
		}
	```
	
	```java
		private final ReentrantLock lock = new ReentrantLock();

		public void lockUpdate() {
			lock.lock(); // Explicitly acquire lock
			try {
				// Critical section
				System.out.println("Locked with ReentrantLock");
			} finally {
				lock.unlock(); // Must release in finally block
			}
		}

	```

-	** When would you NOT use synchronized ? **
	-	You should NOT use synchronized when accessing single-threaded code, handling read-heavy data, or when performance is critical, as it causes blocking, thread contention, and potential deadlocks.
	-	Avoid it for read-only operations, simple volatile variable updates, or when using more flexible utilities from the java.util.concurrent package, such as Lock or Atomic classes
	
-	** What will happen when we call start() two times on the same thread ? **
	-	Throws Exception: It will throw an IllegalThreadStateException.
	-	Reason: In Java, a thread can only be started once. Once a thread has finished its execution (reaches the Terminated state), it cannot be restarted.
	-	Internal Check: The JVM maintains a threadStatus variable; the start() method checks this status at the beginning and throws the exception if the status is not "New" (0).
	
-	** What is Callable in Java threads ? **

-	** What is the need for threads in Java ? **
	-	Parallelism: Allows a program to perform multiple tasks simultaneously on multi-core CPUs, such as processing multiple user requests in a web server.
	-	Better Resource Utilization: Prevents the CPU from sitting idle while waiting for slow I/O operations (like database queries or file reading); another thread can work while one is blocked.

-	** How do you create a thread in Java ? **
	-	Extending the Thread class: Create a subclass, override the run() method, and call start(). This is less flexible because Java doesn't support multiple inheritance.
	-	Implementing the Runnable interface: Define the run() logic in a class, pass it to a Thread constructor, and call start(). This is the preferred approach as it separates the task from the thread.
	-	Implementing the Callable interface: Similar to Runnable but allows the thread to return a result and throw checked exceptions; must be used with an ExecutorService.
	-	Using Lambda Expressions (Java 8+): For short tasks, you can use a lambda to avoid creating a new class: new Thread(() -> System.out.println("Running")).start();.
	
-	** How do you create a thread by extending the Thread class ? **
	-	Step 1: Create a class that extends the java.lang.Thread class.
	-	Step 2: Override the run() method to define the specific code the thread should execute.
	-	Step 3: Create an instance of your class.
	-	Step 4: Call the start() method (not run()) to register the thread with the JVM and begin execution.
	```java

-	** How do you run a thread in Java ? **
	-	The start() Method: To run a thread, you must call its start() method. This is a special method that notifies the JVM to perform system-level tasks—like allocating a stack and a program counter—before automatically calling your run() method in a new execution path.
	-	Why not run()?: If you call run() directly (e.g., myThread.run()), it acts like a normal method call. No new thread is created, and the code executes sequentially in the current thread (usually the main thread), defeating the purpose of multithreading.
	-	The Thread Scheduler: Once start() is called, the thread enters the Runnable state. It doesn't necessarily run immediately; it waits for the OS/JVM Thread Scheduler to pick it and give it CPU time.
	-	One-Time Use: A thread can only be run once. Attempting to call start() again on the same thread instance will throw an IllegalThreadStateException.

-	** What are the different states of a thread ? **
	-	New: The thread is created as an object (e.g., Thread t = new Thread()) but the start() method has not yet been called.
	-	Runnable: The thread is ready to run after start() is called; it is either currently executing or waiting for the OS thread scheduler to allocate CPU time.
	-	Blocked: The thread is alive but suspended because it is waiting to acquire a monitor lock to enter a synchronized block or method.
	-	Waiting: The thread is waiting indefinitely for another thread to perform a specific action, triggered by methods like Object.wait() or Thread.join().
	-	Timed Waiting: The thread is waiting for a specified period before resuming, caused by methods like Thread.sleep(ms), wait(ms), or join(ms).
	-	Terminated: The thread has finished its execution of the run() method or was stopped due to an unhandled exception; it cannot be restarted.

-	** What is ExecutorService ? **
	-	Definition: ExecutorService is a high-level API in the java.util.concurrent package used for managing and controlling thread execution. It simplifies concurrent programming by decoupling task submission from the actual thread management.
	-	Thread Pooling: Instead of manually creating new threads for every task, it maintains a reusable pool of worker threads. This reduces the overhead of thread creation and prevents resource exhaustion.
	-	Task Queueing: If more tasks are submitted than available threads, the service automatically stores them in an internal BlockingQueue until a thread becomes free.
	-	Lifecycle Management: It provides methods to gracefully shut down the service, ensuring that ongoing tasks finish before the JVM exits.
	-	Result Retrieval: Unlike basic threads, it can execute Callable tasks that return a Future object, allowing you to retrieve results or check the status of asynchronous work. 
	-	Common Ways to Create an ExecutorService:
		-	Executors.newFixedThreadPool(n): Creates a pool with a fixed number of threads.
		-	Executors.newCachedThreadPool(): Dynamically creates threads as needed and reuses idle ones.
		-	Executors.newSingleThreadExecutor(): Guarantees all tasks execute sequentially using exactly one thread.
		-	Executors.newScheduledThreadPool(n): Supports running tasks after a delay or periodically.

-	** What are the different ways of creating Executor Services ? **
	-	Standard Factory Methods (Executors class): The most common way is using static factory methods that provide pre-configured thread pools for different use cases.

		-	Usage: Executors.newVirtualThreadPerTaskExecutor().
		-	Direct Constructor (ThreadPoolExecutor): For maximum control, you can manually instantiate a ThreadPoolExecutor class, allowing you to configure parameters like core/max pool size, keep-alive time, and the BlockingQueue type.

	-	** What is synchronization of threads ? **
		-	Definition: Synchronization is a mechanism in Java that ensures only one thread can access a shared resource (like an object, variable, or file) at any given time.
		-	The Problem (Race Condition): Without synchronization, if two threads try to modify the same data simultaneously, the data can become inconsistent or corrupted.
		-	The Monitor (Lock): Every object in Java has an internal "lock" called a monitor. When a thread enters a synchronized section, it "acquires" this lock. Other threads must wait until the lock is released.
		-	Types of Synchronization:
			-	Synchronized Method: Locks the entire object instance for the duration of the method call.
			-	Synchronized Block: Locks only a specific part of the code, which is more efficient as it reduces the time other threads are kept waiting.
			-	Static Synchronization: Locks the Class object itself, preventing multiple threads from accessing static resources simultaneously.
		```java
		class SafeCounter {

			// Fixed with synchronized
			public synchronized void increment() {
				count++;
			}

			public int getCount() {
				return count;
			}
		}

		public class Main {
			public static void main(String[] args) throws InterruptedException {
				SafeCounter sc = new SafeCounter();
				
				Thread t1 = new Thread(() -> { for(int i=0; i<10000; i++) sc.increment(); });
				Thread t2 = new Thread(() -> { for(int i=0; i<10000; i++) sc.increment(); });

				t2.start();
				t1.join();
				t2.join();

-	** Can you provide an example of a synchronized block ? **
	-	Implementation: Instead of locking the whole method, you wrap only the specific critical line of code inside synchronized(this). This allows other non-critical parts of the method to run in parallel, improving performance.
	-	Key Advantage: It reduces lock contention. If the method had 100 lines of code but only 1 line modified a shared variable, a synchronized block ensures threads only wait for that 1 line instead of the entire 100.
	-	```java

				private int count = 0;

				public void addOne() {
					// Non-critical code (runs in parallel)
					String threadName = Thread.currentThread().getName();
					System.out.println(threadName + " is preparing to increment...");

						count++;

				}

		```

-	** Can a static method be synchronized ? **
	-	Yes: A static method can be synchronized by adding the synchronized keyword to its declaration.
	-	Class-Level Lock: When a static method is synchronized, it acquires a lock on the Class object (ClassName.class) rather than an individual instance (this).
	-	Behavior: This means that only one thread can execute any synchronized static method of that class at a time, across all instances of that class.
	-	Independence: A thread holding a class-level lock for a static method does not prevent other threads from calling non-static synchronized methods on a specific instance of that same class (and vice versa), as they use different locks.
	-	Key Takeaway: Static synchronization is used to protect static data (shared across all objects), while instance synchronization protects instance data (unique to each object).
	```java
	class SyncDemo {
		// Instance synchronization: Locks 'this'
		public synchronized void instanceMethod() {
			System.out.println(Thread.currentThread().getName() + " entered instanceMethod");
			try { Thread.sleep(2000); } catch (InterruptedException e) {}
		}

		// Static synchronization: Locks 'SyncDemo.class'
		public static synchronized void staticMethod() {
			System.out.println(Thread.currentThread().getName() + " entered staticMethod");
			try { Thread.sleep(2000); } catch (InterruptedException e) {}
		}
	}

	public class Main {
		public static void main(String[] args) {
			SyncDemo obj1 = new SyncDemo();
			SyncDemo obj2 = new SyncDemo();

			// These run IN PARALLEL because they lock different instances (obj1 and obj2)
			new Thread(() -> obj1.instanceMethod(), "Thread-1").start();
			new Thread(() -> obj2.instanceMethod(), "Thread-2").start();

			// These run SEQUENTIALLY because they both try to lock the same Class object
			new Thread(() -> SyncDemo.staticMethod(), "Thread-3").start();
			new Thread(() -> SyncDemo.staticMethod(), "Thread-4").start();
		}
	}

	```

-	** What are some other important methods in the Thread class ? **

	-	Thread.yield(): A hint to the thread scheduler that the current thread is willing to give up its current use of the processor so other threads of the same priority can run.
	-	join(): Allows one thread to wait for the completion of another. If t1.join() is called in the main thread, the main thread stops until t1 finishes.

-	** What are the important methods in Java for inter-thread communication ? **
	-	wait(): Causes the current thread to wait and release the lock on the object it is synchronised on. It remains in a waiting state until another thread calls notify() or notifyAll() on the same object.
	-	notify(): Wakes up a single thread that is waiting on that object's monitor. If multiple threads are waiting, the JVM picks one at random.
	-	notifyAll(): Wakes up all threads that are waiting on that object's monitor. This is generally safer than notify() as it ensures all dependent tasks have a chance to proceed.
	-	Requirements: All three methods must be called from within a synchronized context (block or method). If called without holding the monitor lock, they throw an IllegalMonitorStateException.
	-	Usage Pattern: wait() should always be used inside a while loop to check the condition, preventing "spurious wakeups" where a thread wakes up without being signaled.

-	** Can you provide a code example using wait() and notify() ? **
	-	Implementation: This is the classic Producer-Consumer pattern. The Producer adds data and calls notify(), while the Consumer waits using wait() if the queue is empty. Both must synchronize on the same shared object.
	-	Key Rule: The wait() method releases the lock so the other thread can enter its synchronized block, whereas Thread.sleep() keeps the lock.
	```java
		class SharedQueue {
			private int data;
			private boolean hasData = false;

			// Consumer calls this
			public synchronized void consume() {
				while (!hasData) { // Always use while loop for wait()
					try { wait(); } catch (InterruptedException e) {}
				}
				System.out.println("Consumed: " + data);
				hasData = false;
				notify(); // Wake up the producer
			}

			// Producer calls this
			public synchronized void produce(int value) {
				while (hasData) {
					try { wait(); } catch (InterruptedException e) {}
				}
				this.data = value;
				hasData = true;
				System.out.println("Produced: " + value);
				notify(); // Wake up the consumer
			}
		}

		public class WaitNotifyDemo {
			public static void main(String[] args) {
				SharedQueue queue = new SharedQueue();
				new Thread(() -> { for(int i=1; i<=5; i++) queue.produce(i); }).start();
				new Thread(() -> { for(int i=1; i<=5; i++) queue.consume(); }).start();
			}
		}
	```

-	** Why is notifyAll() generally preferred over notify() ? **
	-	The "Lost Signal" Problem: notify() only wakes up one random thread. If that thread can't proceed (e.g., a specific condition isn't met), it goes back to sleep, but other threads that could have proceeded remain stuck waiting forever.
	-	Multiple Conditions: In a system with different types of waiters (e.g., multiple producers and multiple consumers), notify() might accidentally wake a producer when it should have woken a consumer. notifyAll() ensures every thread checks its condition.
	-	Livelock Prevention: Using notifyAll() is a safer "catch-all" that prevents scenarios where the entire system hangs because the "wrong" thread was signaled.
	-	Performance Myth: While notifyAll() wakes everyone, only one thread can actually re-acquire the lock at a time. The others will immediately block again, so the performance overhead is usually negligible compared to the risk of a deadlocked system.

> ** Locking Mechanisms: ** Deep knowledge of mutexes, conditional variables, and preventing deadlocks is crucial

-	** How do you prevent deadlocks in Java ? **
	-	Lock Ordering: Always acquire multiple locks in the same predefined order across all threads. if Thread A and Thread B both try to get Lock 1 then Lock 2, they will never block each other in a cycle.
	-	Lock Timeout (tryLock): Use ReentrantLock.tryLock(timeout, unit) instead of synchronized blocks. This allows a thread to give up and release its own held locks if it can't acquire the next one within a specific timeframe.
	-	Deadlock Detection: Use tools like jstack or jconsole to monitor thread dumps; the JVM can detect deadlocks and identify which threads and monitors are involved in the "circular wait."
	-	Avoid Nested Locks: Design your code to require only one lock at a time whenever possible. If you must use multiple, encapsulate them within a single higher-level object to hide the complexity.
	-	Use Atomic Variables: Replace locks entirely with AtomicInteger or ConcurrentHashMap for simple state management, as these use lock-free algorithms (CAS) that cannot result in a deadlock.
	
-	** Can you provide a code example of how Lock Ordering prevents deadlocks ? **
	-	The Problem: If Thread 1 locks A then B, and Thread 2 locks B then A, they can get stuck in a "circular wait."
	-	The Solution: Force both threads to acquire locks in the exact same order (e.g., always Lock A first, then Lock B).
	-	Result: Thread-2 will simply wait for Thread-1 to release Lock A before it even tries to touch Lock B, completely eliminating the possibility of a circular dependency.
	```java
		Object lockA = new Object();
		Object lockB = new Object();

		// Both threads follow the same "A then B" rule
		Runnable task = () -> {
			synchronized (lockA) { // Always first
				System.out.println(Thread.currentThread().getName() + " acquired Lock A");
				try { Thread.sleep(50); } catch (InterruptedException e) {}

				synchronized (lockB) { // Always second
					System.out.println(Thread.currentThread().getName() + " acquired Lock B");
				}
			}
		};

		new Thread(task, "Thread-1").start();
		new Thread(task, "Thread-2").start();
	```

** Can you provide a Producer-Consumer code example using BlockingQueue ? **
	-	Implementation: The BlockingQueue is the most efficient way to solve this problem because it handles thread signaling (wait/notify) and locking internally. The producer blocks if the queue is full, and the consumer blocks if the queue is empty
	-	Key Advantage: Using a BlockingQueue eliminates the need for manual synchronized blocks, wait(), and notify(), making the code much cleaner and less prone to concurrency bugs.
	```java
		import java.util.concurrent.ArrayBlockingQueue;
		import java.util.concurrent.BlockingQueue;

		public class ProducerConsumer {
			public static void main(String[] args) {
				// Thread-safe queue with a capacity of 5
				BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(5);

				// Producer Thread
				new Thread(() -> {
					try {
						for (int i = 1; i <= 10; i++) {
							queue.put(i); // Blocks if queue is full
							System.out.println("Produced: " + i);
							Thread.sleep(500); // Simulate production time
						}
					} catch (InterruptedException e) { Thread.currentThread().interrupt(); }
				}).start();

				// Consumer Thread
				new Thread(() -> {
					try {
						while (true) {

** How do you implement a custom Thread Pool in Java ? **
	-	Concept: A custom thread pool reuses a fixed set of threads to execute tasks from a shared queue, avoiding the overhead of creating and destroying threads for every operation.
	-	Core Components:
		-	Task Queue: A BlockingQueue<Runnable> (like LinkedBlockingQueue) that holds submitted tasks until a thread is free.
		-	Worker Threads: A set of threads that continuously poll the queue and run tasks in a while(true) loop.
		-	Submit Method: A public method to add new Runnable tasks to the internal queue.
	-	Refinement: Professional implementations like ThreadPoolExecutor add features like shutdown (to stop threads gracefully), timeouts for idle threads, and rejection policies for when the queue is full	
	```java
		public class MyThreadPool {
			private final BlockingQueue<Runnable> taskQueue;
			private final List<Worker> workers = new ArrayList<>();

			public MyThreadPool(int numThreads) {
				taskQueue = new LinkedBlockingQueue<>();
				for (int i = 0; i < numThreads; i++) {
					Worker worker = new Worker();
					workers.add(worker);
					worker.start(); // Start the persistent workers
				}
			}

			public void execute(Runnable task) {
				taskQueue.add(task); // Add task to the shared queue
			}

			private class Worker extends Thread {
				public void run() {
					while (true) {
						try {
							Runnable task = taskQueue.take(); // Wait for a task
							task.run(); // Execute it
						} catch (InterruptedException e) { break; }
					}
				}
			}
		}
	```

-	** How do you execute dependent tasks in parallel ? **
	-	Conceptual Approach: You model the tasks as a Directed Acyclic Graph (DAG). Independent tasks are executed simultaneously, while dependent tasks are triggered automatically as soon as all their "parent" tasks complete.
	-	Using CompletableFuture (Java 8+): This is the built-in way to handle parallel dependencies.
	-	Parallel Roots: Launch multiple supplyAsync() calls for tasks with no dependencies.
	-	Combining (AND): Use thenCombine() or allOf() to wait for multiple tasks to finish before starting a downstream one.
	-	Chaining (Serial within Parallel): Use thenApply() or thenCompose() for steps that must follow one another within a specific parallel branch.
	-	Using CountDownLatch: A lower-level alternative where a downstream task calls await() and is released only after all independent upstream tasks call countDown().
	-	The Scenario:
		-	Task A and Task B are independent (run in parallel).
		-	Task C depends on A.
		-	Task D depends on B.
		-	Task E (Final) depends on both C and D.
		-	Benefit: This model ensures that while A and B run together, and C and D run together, the final step E automatically waits for all prerequisites without manual if checks or wait/notify logic.
		```java
			// 1. Start independent tasks in parallel
			CompletableFuture<String> taskA = CompletableFuture.supplyAsync(() -> {
				System.out.println("Task A: Fetching User...");
				return "User1";
			});

			CompletableFuture<String> taskB = CompletableFuture.supplyAsync(() -> {
				System.out.println("Task B: Fetching Orders...");
				return "Orders_List";
			});

			// 2. Start dependent child tasks
			CompletableFuture<String> taskC = taskA.thenApply(user -> {
				System.out.println("Task C: Validating " + user);
				return user + "_Validated";
			});

			CompletableFuture<String> taskD = taskB.thenApply(orders -> {
				System.out.println("Task D: Calculating total for " + orders);
				return "Total_$500";
			});

			// 3. Final aggregation (wait for C and D)
			CompletableFuture<String> taskE = taskC.thenCombine(taskD, (userVal, total) -> {
				System.out.println("Task E: Generating Final Invoice...");
				return "Invoice for " + userVal + ": " + total;
			});

			// Block and print result
			System.out.println("Result: " + taskE.join());

		```
		
-	** Can you explain the dining philosophers problem ? **  
	-	The dining philosophers problem is a classic computer science synchronization puzzle illustrating challenges in concurrent algorithm design, such as deadlock and starvation.
	-	Five philosophers sit at a circular table, alternating between thinking and eating, requiring two forks (shared resources) to eat. To prevent deadlock, algorithms must ensure all philosophers can eat without causing circular wait, such as limiting diners or regulating fork acquisition.
	
	```java
	import java.util.concurrent.Semaphore;

	class Philosopher extends Thread {
		private int id;
		private Semaphore leftFork;
		private Semaphore rightFork;

		public Philosopher(int id, Semaphore left, Semaphore right) {
			this.id = id;
			this.leftFork = left;
			this.rightFork = right;
		}

		private void performAction(String action) throws InterruptedException {
			System.out.println("Philosopher " + id + " is " + action);
			// Simulate time spent thinking or eating
			Thread.sleep(((int) (Math.random() * 100)));
		}

		@Override
		public void run() {
			try {
				while (true) {
					performAction("thinking");
					
					// DEADLOCK PREVENTION: 
					// Asymmetric solution: Even-numbered philosophers pick up left then right,
					// while odd-numbered philosophers pick up right then left.
					if (id % 2 == 0) {
						leftFork.acquire();
						rightFork.acquire();
					} else {
						rightFork.acquire();
						leftFork.acquire();
					}

					performAction("eating");

					// Put down forks
					leftFork.release();
					rightFork.release();
				}
			} catch (InterruptedException e) {
				Thread.currentThread().interrupt();
			}
		}
	}

	public class DiningPhilosophers {
		public static void main(String[] args) {
			final int numPhilosophers = 5;
			Semaphore[] forks = new Semaphore[numPhilosophers];

			// Initialize each fork as a semaphore with 1 permit (available)
			for (int i = 0; i < numPhilosophers; i++) {
				forks[i] = new Semaphore(1);
			}

			// Create and start philosopher threads
			for (int i = 0; i < numPhilosophers; i++) {
				Semaphore leftFork = forks[i];
				Semaphore rightFork = forks[(i + 1) % numPhilosophers];
				new Philosopher(i, leftFork, rightFork).start();
			}
		}
	}

	```

## Java Strings

-	** Why is String Immutable? **
	-	https://www.youtube.com/watch?v=to9DPVsdByE
	-	thread safe & to avoid garbage collector to hold value in string pool for security reasons, synchronization and concurrency, caching, and class loading
	-	String Pool (Caching): If Strings were mutable, changing one reference would affect all other variables pointing to the same literal in the pool.
	-	Security: Strings are used for sensitive data like Database URLs, Usernames, and Passwords. If they were mutable, a malicious process could change the value after validation.
	-	Thread Safety: Since a String cannot change, it is inherently thread-safe. Multiple threads can share the same instance without any synchronization or risk of data corruption.
	-	Class Loading: Java class loaders use Strings to load classes. If a String were mutable, a hacker could change the name of a class (e.g., java.io.File) to point to a malicious class during the loading process.
	-	Hash Code Caching: Because the content is fixed, the hashCode is calculated once and cached. This makes Strings very fast when used as keys in a HashMap.

-	** How do you create an Immutable Class in Java? **

-	** What are the limitations of Java Records? **
	-	No Inheritance: Records cannot extend any other class because they already implicitly extend java.lang.Record. Since Java doesn't support multiple inheritance, they are locked into this hierarchy.
	-	Implicitly Final: A record is always final, meaning you cannot create a subclass of a record.
	-	No Instance Fields: You cannot add extra instance variables (fields) inside the record body; all data must be defined in the header (the parenthesis). You can, however, add static fields.
	-	Shallow Immutability: While the record's references are final, the objects they point to might not be. If a record holds a standard ArrayList, a caller can still modify the contents of that list unless you wrap it in an unmodifiable collection.
	-	No Argument-less Constructor: By default, records only provide the canonical constructor (the one matching the header). If you need a no-args constructor, you must write it manually and call this(...).
	-	| Feature            | Regular Class                | Record                     |
		| ------------------	| ----------------------------	| --------------------------	|
		| Boilerplate        | High (Getters, equals, etc.) | None (Auto-generated)      |
		| Extends Classes    | Yes                          | No                         |
		| Final by Default   | No                           | Yes                        |
		| Field Modification | Possible (if not final)      | Impossible (All are final) |

-	** What is StringTokenizer in Java? **
	-	Purpose: StringTokenizer is a legacy utility class in the java.util package explicitly used to split strings into smaller parts called tokens based on specific characters (delimiters).
	-	Legacy Status: Introduced in JDK 1.0, it is now considered outdated. For modern applications, it is generally recommended to use String.split() or the Scanner class instead.
	-	```java
		import java.util.StringTokenizer;

		public class TokenDemo {
			public static void main(String[] args) {
				String data = "Java,Python,C++,JavaScript";
				// Splitting by comma
				StringTokenizer st = new StringTokenizer(data, ",");
				
				System.out.println("Token count: " + st.countTokens());
				while (st.hasMoreTokens()) {
					System.out.println(st.nextToken());
				}
			}
		}
		```

-	** How do you use String.split() for tokenization? **
	-	Purpose: It is the modern standard for splitting strings. It uses Regular Expressions (regex) to find delimiters and returns an array of all parts.
	-	Key Feature: Unlike StringTokenizer, it can split by complex patterns (e.g., split by any digit or split by one or more spaces).
	-	Overloaded Method: split(regex, limit) allows you to control how many times the split occurs. A limit of 2 means it splits at the first match and keeps the rest of the string as the second element.
	-	Performance: If you are splitting a very large string by a single character (like ,) millions of times, StringTokenizer is slightly faster because it doesnt compile a regex.
	-	Flexibility: For almost all other cases, String.split() is better because it is more readable and handles multiple whitespace variations easily.
	-	```java
		public class SplitDemo {
			public static void main(String[] args) {
				String data = "Java   Python,C++:JavaScript";

				// 1. Split by a simple comma
				String[] simple = data.split(","); 

				// 2. Split by multiple delimiters (comma, colon, or space)
				// [ ,:]+ means "any of these characters, one or more times"
				String[] complex = data.split("[ ,:]+");

				for (String lang : complex) {
					System.out.println("Language: " + lang);
				}

				// 3. Using a limit
				String[] limited = "a:b:c:d".split(":", 2);
				// Result: ["a", "b:c:d"]
			}
		}
		```

-	**What is the difference between equals() , == , and compareTo() in Java ? **
	-	==: An operator used for reference comparison; it checks if two variables point to the exact same memory address.
	-	equals(): A method used for content comparison; it checks if two objects are logically equivalent based on their data. Things to consider: You must override hashCode() if you override equals(), and always handle potential NullPointerException by using Objects.equals(a, b).
	-	compareTo(): A method from the Comparable interface used for natural ordering; it returns an integer (negative, zero, or positive) to indicate if one object is less than, equal to, or greater than another for sorting purposes.
